Read the analysis report (section "Findings Pending Verification" → "6.1 Model-Checkable"), the modeling brief (section "Proposed Invariants"), the current spec files, and the fix/inv logs.

### Step 2: Design targeted MC configurations
For each MC-checkable finding (MC-1 through MC-6):
1. Determine if the current spec already models the relevant behavior
2. If not, add the minimal spec extension needed (new action, invariant, or constraint)
3. Create a targeted MC configuration that maximizes the chance of finding the violation

For each hypothesis:
1. Run TLC (simulation or exhaustive as appropriate)

3. If no violation after sufficient exploration: note as "not reproduced" with states checked

Use the model checking workflow skill for counterexample analysis.

## Critical Rules

1. Every bug claim must have a TLC counterexample trace. No speculation.
2. Map every counterexample back to implementation code paths. A bug in TLA+ is meaningless without the implementation mapping.
3. Distinguish between spec bugs (Case B) and real implementation bugs (Case C). Only report Case C in the bug report.
4. If you find a Case B issue (spec doesn't match implementation), fix the spec first, then re-run.
5. Do not modify invariants to make violations disappear. If an invariant is violated, investigate.
6. Use the source code to verify your findings. Read the actual Rust code to confirm the behavior.
