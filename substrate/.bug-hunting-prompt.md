## Validation-Phase Notes

During validation, two invariants were removed as too strong (Case A):
- `VoteLimitRespected` — removed because forced changes can advance finalization past pending standard change limits
- `StandardChangeOrdering` — removed because forced changes can legitimately skip standard change order

MC-1 targets VoteLimitRespected which was removed. You need to design a more precise version that only applies when no forced changes are active.

## Task

### Step 1: Read context
Read the analysis report, modeling brief, changelog, current spec files.

### Step 2: Design targeted MC configurations
For each MC-checkable finding (MC-1 through MC-8):
1. Determine if the current spec already models the relevant behavior
2. If not, add the minimal spec extension needed
3. Create a targeted MC configuration

Strategies:
- **Increase bounds** where needed (more blocks, more rounds)
- **Use simulation mode** for large state spaces
- **Add targeted invariants** for each hypothesis
- **Model equivocation** for MC-5/MC-6 (Byzantine node casting conflicting votes)
- **Model concurrent finalization paths** for MC-4 (gossip vs sync)
- **Model crash between HasVoted write and round completion** for MC-7

### Step 3: Run model checking
For each hypothesis:
1. Run TLC (simulation or exhaustive as appropriate)
2. If violation found: analyze counterexample, trace to implementation code, classify
3. If no violation: note as "not reproduced" with states checked

## Critical Rules

1. Every bug claim must have a TLC counterexample trace. No speculation.
2. Map every counterexample back to implementation code paths.
3. Distinguish between spec bugs (Case B) and real implementation bugs (Case C). Only report Case C.
4. If you find a Case B issue, fix the spec first, then re-run.
5. Do not modify invariants to make violations disappear.
6. Use the source code to verify findings. Read the actual Rust code to confirm.
