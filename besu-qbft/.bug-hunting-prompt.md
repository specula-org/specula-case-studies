## Key Code-Level Hints

- **QbftController.java:261-271 vs 276-289**: Block timer expiry lacks the `blockchain.getChainHeadBlockNumber()` guard that round expiry has. Core of MC-1.
- **RoundChangeManager.java:52,69,90**: `actioned` one-shot flag prevents certificate re-creation. Core of MC-2.
- **RoundChangeManager.java:156 vs 70**: `roundSummary` uses `put()` (overwrites) while `roundChangeCache` uses `putIfAbsent()` (preserves first). Core of MC-3.
- **RoundChangeArtifacts.java:72-85**: Comparator violates antisymmetric contract when both operands have empty `PreparedRoundMetadata`. Core of MC-6.
- **Historical safety violation #1734**: Two honest IBFT2 validators committed different blocks at same height due to event queue race.

## Task

### Step 1: Read context
Read the analysis report, modeling brief, current spec files, and fix/inv logs.

### Step 2: Design targeted MC configurations
For each MC-checkable finding (MC-1 through MC-6):
1. Determine if the current spec already models the relevant behavior
2. If not, add the minimal spec extension needed
3. Create a targeted MC configuration

Strategies:
- **Use 4 servers** (f=1) for byzantine tolerance tests
- **Increase MaxRound** for round change related bugs (MC-2, MC-3, MC-6)
- **Use simulation mode** for large state spaces
- **Add targeted invariants** for each hypothesis
- **Model the asymmetric guard** explicitly for MC-1 (block timer vs round expiry)

### Step 3: Run model checking
For each hypothesis:
1. Run TLC (simulation or exhaustive as appropriate)
2. If violation found: analyze counterexample, trace to implementation code, classify
3. If no violation: note as "not reproduced" with states checked

## Critical Rules

1. Every bug claim must have a TLC counterexample trace. No speculation.
2. Map every counterexample back to implementation code paths.
3. Distinguish between spec bugs (Case B) and real implementation bugs (Case C). Only report Case C.
4. If you find a Case B issue, fix the spec first, then re-run.
5. Do not modify invariants to make violations disappear.
6. Use the source code to verify findings. Read the actual Java code to confirm.
