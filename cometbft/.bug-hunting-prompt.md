## Task

### Step 1: Read context
Read the analysis report (section "Findings Pending Verification" → "6.1 Model-Checkable"), the modeling brief (section "Proposed Invariants"), the current MC.tla/MC.cfg, and the fix_log.md to understand what's already been checked.

### Step 2: Design targeted MC configurations
For each MC-checkable finding (MC-1 through MC-10):
1. Determine if the current spec already models the relevant behavior
2. If not, add the minimal spec extension needed (new action, invariant, or constraint)
3. Create a targeted MC configuration that maximizes the chance of finding the violation

Strategies:
- **Increase bounds** where needed (e.g., MaxRound=3 for round-skip bugs, MaxHeight=3 for cross-height bugs)
- **Use simulation mode** (`-simulate`) for large state spaces
- **Add targeted invariants** that capture the specific expected violation
- **Reduce irrelevant state** by disabling unrelated extensions when testing a specific family
- **Use 4 servers** (Server = {s1, s2, s3, s4}) where f=1 matters (byzantine tolerance tests)

### Step 3: Run model checking
For each hypothesis:
1. Run TLC (simulation or exhaustive as appropriate)
2. If violation found: analyze the counterexample, trace it back to the implementation code, classify as real bug vs spec issue
3. If no violation after sufficient exploration: note as "not reproduced" with states checked

Use the model checking workflow skill for counterexample analysis.

## Critical Rules

1. Every bug claim must have a TLC counterexample trace. No speculation.
2. Map every counterexample back to implementation code paths. A bug in TLA+ is meaningless without the implementation mapping.
3. Distinguish between spec bugs (Case B) and real implementation bugs (Case C). Only report Case C in the bug report.
4. If you find a Case B issue (spec doesn't match implementation), fix the spec first, then re-run.
5. For known open bugs (B1, B6, B15): reproducing them with a counterexample is valuable — it confirms the spec captures the bug.
6. Do not modify invariants to make violations disappear. If an invariant is violated, investigate.
7. Use the source code to verify your findings. Read the actual Go code to confirm the behavior.
